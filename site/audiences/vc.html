なぜ 税務会計AI か

法人番号 1 リクエスト で 4 dataset 横断

enforcement + cases + invoice + programs

13 桁の法人番号 (国税庁公表) を投げるだけで、行政処分・採択履歴・適格事業者登録・関連制度を統一スキーマで返却。社内 DD ツールに REST or MCP で統合。

一次資料のみ、出典 URL 必須

集約サイト不採用

行政処分は各監督官庁公表分、採択は経産省・農水省・中小企業庁・各自治体公表分、適格事業者は国税庁 PDL v1.0。集約サイト (二次情報源) は使わない。各行に source_url + fetched_at 付き。

利用量制限なし、¥3/リクエスト 一律

月額固定・seat なし

月 1M req でも単価は ¥3 (税込 ¥3.30) のまま。最低金額・年間契約・seat fee なし。Stripe メータード請求で運用 scalable、適格請求書 (T8010001213708) 自動発行。

なぜ VC がこれを使うべきか — IRR の話

  • ポートフォリオ会社が取りこぼしている補助金 = 事業外収益。100社 × 平均1件 × 平均¥500万 = ¥5億の取りこぼし。1年で IRR を 1-2pt 改善する原資。
  • DD で「未受給の優良制度」を発見 = 投資後 100 日プランの quick win。経営チームの信頼獲得に直結。
  • 行政処分・ネガ判明 = 投資判断の事前回避。1案件あたり数千万〜数億の毀損回避はファンド全体の IRR を守る。

主要 use case

1. 投資候補の compliance check

シードラウンドの 50 社を法人番号 一括処理 投入 → 行政処分歴 (建設業法・金商法・電気通信事業法等)、適格請求書登録の有無、関連補助金採択歴を一覧化。投資判断会議の事前資料を自動生成。

2. M&A buy-side DD: 過去 5 年の行政処分

買収対象 + 主要取引先 30 社の法人番号で行政処分歴を 1 一括処理。check_enforcement_am tool で重大処分 (営業停止・登録取消) を即抽出、レポート添付。

3. 採択歴で技術検証

NEDO・経産省ものづくり補助金・中小企業庁事業再構築の採択履歴は技術力の客観指標。search_acceptance_stats_am で業種別採択率 + 当該社の採択回数を取得。

4. ポートフォリオ会社の支援可能制度

投資後の value-up 文脈で、ポートフォリオ会社が使える補助金・税制特例・融資を 1 リクエスト で抽出。排他ルール 181 本で「これとこれは併用不可」を事前検出。

5. インボイス対応 DD

取引先・仕入先の適格請求書発行事業者登録を法人番号で確認。invoice_registrants 13,801 件 (PDL v1.0 attribution、月次フル更新準備中) で登録日・業態を照会。

houjin_bangou は 13 桁の国税庁発番号。/v1/am/enforcement は法人ピンポイント検索用、/v1/enforcement-cases/search は会計検査院公表に法人番号が含まれない (recipient_houjin_bangou 100% NULL) ため期間・省庁絞り込み用。

Python 一括処理 (商用 ¥3/リクエスト)

# pip install httpx
import httpx, asyncio

# DD 標準パッケージ: 法人番号 1 件で 3 endpoint を chain、複数法人を並列実行
nums = ["8010001213708", "1010001067912"]  # 13 桁の国税庁発番号
BASE = "https://api.zeimu-kaikei.ai/v1"

async def dd(client, num):
    # 適格請求書発行事業者登録 (法人番号は path "T<13桁>" 形式)
    invoice = await client.get(f"{BASE}/invoice_registrants/T{num}")
    # 行政処分・排除期間 状態 (法人ピンポイント、houjin_bangou + as_of_date)
    enforce = await client.get(f"{BASE}/am/enforcement",
                               params={"houjin_bangou": num,
                                       "as_of_date": "2026-04-29"})
    # 補助金採択履歴 (case_studies の houjin_bangou 付与は約 19% — 不明な場合は q=社名 推奨)
    cases = await client.get(f"{BASE}/case-studies/search",
                             params={"houjin_bangou": num, "limit": 20})
    return {"invoice": invoice.json(), "enforcement": enforce.json(),
            "case_studies": cases.json()}

async def main():
    async with httpx.AsyncClient(headers={"X-API-Key": "sk_..."}) as client:
        results = await asyncio.gather(*[dd(client, n) for n in nums])
    for r in results:
        print(r)

asyncio.run(main())

3 API 経路 chain。100 社並列、hard cap なし。法人番号 1 件 = 3 リクエスト = ¥9 (税別)。/v1/am/enforcement は houjin_bangou 必須 (corporate_number は受け付けない)。case-studies/search は 法人番号付与 19% のため q=社名 併用も推奨。

honest data disclosure / 業務範囲制限 (§52)

本サービスは税務助言・法律事務を行いません

税務会計AI は公的機関が公表する制度・行政処分・採択結果・適格請求書登録などの一次資料を検索・整理・構造化して提供する 検索インデックス + API/MCP です。税理士法 §52 に基づき個別具体的な税務判断・申告書作成代行は行いません。弁護士法 §72 に基づき法律事務 (法律相談・代理行為) も行いません。DD レポートの判断・投資判断・買収判断は、資格を有する税理士・弁護士・公認会計士・中小企業診断士に必ず確認してください。

本ページに記載する数字 (1,185 行政処分 / 13,801 適格事業者 / 2,286 採択事例 等) は 2026-04-29 時点の DB 行数です。データ鮮度ダッシュボード から live で確認できます。

DD で扱う数字は出所の確からしさを必ず分けて開示します。「sourced (一次資料に直接突合済)」と「heuristic (機械推論で補完、status=unknown フラグ付)」の二段構成です:

heuristic 行は、 例えば「この制度 X とこの制度 Y は併用できますか?」を聞かれた際に未確定として開示する設計です。「sourced と heuristic を混ぜて 4.8万件あります」と言うのは 誇大表現のため、API レスポンスでも明示分離 (status='unknown' フラグ + data_quality.compat_unknown_bucket_pct 添付) してあります。

honest 制限事項

  • enforcement_cases.recipient_houjin_bangou は 100% NULL。会計検査院公表に法人番号は含まれない。法人ピンポイント検索は /v1/am/enforcement?houjin_bangou=... 側 (行政処分詳細 22,258 件、6,455 件法人番号付き) を使用。/v1/enforcement-cases/search は省庁・期間・処分種別フィルタ用。
  • case_studies の法人番号付与は 19% (427 / 2,286)。多くの 採択 公表は 社名のみ。法人番号で hit しない場合は q=<社名> で部分一致検索を併用。
  • case_studies の amount_received_man_yen は付与率 <1% (4 / 2,286)。多くの省庁は 採択 公表時に 交付額を出さない。min_subsidy_yen / max_subsidy_yen は使うと 99% drop。
  • 金額条件 records は外部に集計を出さない (35,713 行のうち 76% が 1 回の不具合のあったデータ取込データ取込からの template-default ¥500K/¥2M 値、再検証中)。個別の DD レポートに「平均交付額 ¥◯◯◯ 万円」として持ち込むのは禁止。
  • laws は本文 154 件 + メタ 9,484 件。e-Gov CC-BY、本文ロード継続中。条文単位の本文照合は本文 154 件のみ可能。
  • 個人 PII / 取引履歴 / 財務情報 は含みません。商業 DB (TDB / TSR) との組合せ運用が前提。

honest 数値は facts.html および transparency.html で SQL 検証可能。誇大記述・出典齟齬を発見されたら info@bookyou.net へご連絡ください、48 時間以内 (営業日) に修正します。

よくある質問 (VC / M&A 専門家)

DD 処理経路 で 100 社並列に投げられますか?

はい。¥3/リクエスト 完全従量 で hard cap なし、スパイク時も 429 になりません。100 社 × 5 API 経路 = 500 リクエスト → ¥1,500 (税別) で 1 一括処理 完了します。

行政処分歴は何年遡れますか?

enforcement_cases 1,185 件 (建設業法・宅建業法・産廃処理法・金商法・電気通信事業法等)。原則 5 年遡及、自治体公開分は遡及期間が異なる場合あり。各行に source_url + fetched_at 付き。

採択歴はどう取得しますか?

case_studies 2,286 件 (補助金採択公表分) を法人番号・名称で照会。10 年程度遡及可能 (制度ごとに公表期間が異なる)。search_acceptance_stats_am で業種・地域別の採択率も確認できます。

適格請求書発行事業者の確認は?

invoice_registrants 13,801 件 delta-only (PDL v1.0 attribution、国税庁公表 一括 pending)。法人番号で登録有無 + 登録日 + 業態を照会。インボイス対応の DD 観点で必須。

scale 制限はありますか?

大規模利用 でも料金は ¥3/リクエスト 一律 完全従量。月額固定・seat fee・年間最低なし。月 1M req でも単価は変わりません。Stripe メータード請求で運用 scalable。

PII / 取引履歴は含まれますか?

税務会計AI は公的開示された一次資料 (行政処分公表・採択結果公表・適格事業者登録 PDL v1.0) のみを扱います。個人 PII・取引履歴・財務情報は含みません。商業 DB との組合せ運用が前提です。

DD レポート自動化に組み込めますか?

はい。MCP サーバ を Claude / GPT に直結 + REST API 経路 を社内 RAG に統合する 2 系統。dd_profile_am tool が法人番号 1 件で標準 DD パッケージを返します。

関連 利用者層 は?

投資先 / 売主側で重なる stakeholders: 税理士 (税務 DD)、行政書士 (許認可承継 DD)、AI 開発者 (社内 DD bot を組む側)、中小企業経営者 (投資先候補)。5 利用者層 全体を比較。
