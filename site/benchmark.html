jpcite ベンチマーク — Opus 4.7 7-turn vs jpcite tool

5 cohort (税理士 / 会計士 / 行政書士 / 司法書士 / 中小経営者) × 50 query = 250 query で Opus 4.7 7-turn と jpcite 本番 tool chain (search → expand → precomputed_answer → cite) を 1-8 軸 rubric (各 0-10、合計 0-80) で厳密ヘッド・トゥ・ヘッド比較します。 LLM-as-judge は 使いません — 評価は text 類似度 + citation 重複 + 構造特徴のみ、決定的に計算します。

結果サマリ (本日 2026-05-17 スコア — スモーク段階)

Opus 4.7 fixture は手作業生成のため、本日時点では 5/250 件が populate 済み (各 cohort 1 件)。 残り 245 件は operator が Claude Code Max Pro で順次生成中。 スコア比は populate 済みサンプル数で割られているため、フル populate 後に再計算される。

1-8 rubric 軸

  1. Correctness — token-set Jaccard 類似度 (0-10)
  2. Completeness — Opus 必須キーワードの掲載率 (0-10)
  3. Citation — citation hostname の重複率 (0-10)
  4. Currency — source_fetched_at ≥ 2026-01-01 の比率 (0-10)
  5. Depth — tool-call step 数の充実 (0-10)
  6. Concision — 過長応答ペナルティ (0-10)
  7. Actionability — 推奨 endpoint の重複率 (0-10)
  8. Cohort-fit — cohort + tier 一致 (0-10)

再現方法 (誰でもローカルで実行可)

git clone https://github.com/shigetosidumeda-cyber/autonomath-mcp.git
cd autonomath-mcp

# 1. 250 query 確認
yq '.queries | length' data/p5_benchmark/queries_2026_05_17.yaml
# → 250

# 2. jpcite baseline 実行 (NO LLM, 完全決定論)
python scripts/bench/run_jpcite_baseline_2026_05_17.py --mode dry

# 3. Opus 4.7 fixture (operator 手動生成、Claude Code Max Pro 経由)
# 詳細: docs/_internal/P5_BENCHMARK_GROUND_TRUTH_GENERATION_2026_05_17.md

# 4. rubric 採点 (NO LLM-as-judge)
python scripts/bench/score_p5_outputs_2026_05_17.py

# 5. テスト
pytest tests/test_ff3_p5_benchmark.py -v

制約と方針

  • jpcite 本番コードに LLM API import は 絶対禁止 (CLAUDE.md §3、tests/test_no_llm_in_production.py で機械的に強制)。
  • Opus 4.7 fixture は operator が Claude Code Max Pro セッション上で手動生成し、静的 JSON として固定。
  • rubric は LLM-as-judge を使わない — text-similarity / citation-overlap / 構造比較のみ。
  • 採点は決定論的 (同じ入力 → 同じ出力)。再現可能性は fingerprint_sha256 で確認可。
