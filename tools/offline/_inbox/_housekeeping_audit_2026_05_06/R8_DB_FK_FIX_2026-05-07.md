### 1.4 Why the fix is `boot_time: manual`, not boot-applied

SQLite cannot ALTER a column-level FK declaration in place. Two correct
rewrite paths exist:

1. **CREATE TABLE â€¦ AS SELECT swap** â€” copies all 1,966 rows into a fresh
   table with the corrected DDL, drops the old, renames the new. Cheap on
   1,966 rows, but the swap path also rewrites `sqlite_master` and triggers
   index rebuilds.
2. **writable_schema patch** â€” `PRAGMA writable_schema=1`, single
   `UPDATE sqlite_master SET sql=...`, `PRAGMA writable_schema=0`,
   `PRAGMA integrity_check`. No row copy. The schema patch is atomic
   inside SQLite's implicit transaction.

Either path is a write operation against a 12.4 GB file. Production policy
is read-only DB on the operator workstation, and any schema swap belongs in
a maintenance window with backup verification â€” not in `entrypoint.sh Â§4`'s
self-heal loop where a transient failure would block boot.

`scripts/migrations/wave24_193_fix_am_region_fk.sql` therefore declares
`-- boot_time: manual` (per the existing precedent of migration 120 â€”
`120_drop_dead_vec_unifts.sql`). `entrypoint.sh Â§4` skips boot_time-manual
files explicitly (`grep -qi "^-- *boot_time: *manual"`), so adding the
file to source does **not** trigger application during the next deploy.

The `autonomath_boot_manifest.txt` allowlist gate is also empty by default
(2026-05-07 state) â€” even without the boot_time marker, manifest mode would
keep this file out of boot self-heal until an operator explicitly listed it.

### 2.2 Why the fix is skipped (not in this packet)

This is a **data-quality gap**, not a schema-cleanliness defect. Two orthogonal
remediation paths exist:

1. **gBiz delta backfill** â€” pull the 357 corp numbers from a fresh gBiz API
   delta and insert them into `houjin_master`. This is an ETL job, not a
   migration. It belongs in a separate session because:
   - it requires gBiz API quota budgeting and rate-limit handling;
   - it touches `houjin_master_refresh_run` and `houjin_change_history`
     bookkeeping;
   - it overlaps the existing `gbiz_*` ingest scripts and should be wired
     through the established cron substrate, not a one-shot migration.

2. **Quarantine view** â€” leave the orphans in place and surface them through
   a `_quarantine` view so callers see the join gap explicitly. This is a
   schema-only addition, but it does not resolve the underlying data gap;
   it only papers over the symptom.

Production self-heal is the cleanest path â€” the next gBiz delta cron run
(`scripts/cron/` â€” gBiz refresh cohort) is expected to converge most of the
357 absent corp numbers, since they correspond to active program adoptions
(real corps that have actually applied for and received public-program
funding). Operator-side intervention is reserved for the residual cohort
