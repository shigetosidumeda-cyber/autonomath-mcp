# R8 — DR Drill End-to-End (2026-05-07)

Verification pass for the freshly-authored
`docs/runbook/db_corruption_recovery.md` (G1) runbook. The runbook had not
been exercised once before this drill; this report is the first end-to-end
walk-through that times each step on real artifacts and confirms the failure
detector trips on a deliberately-mangled backup.

This is **internal-hypothesis framing**: the drill runs against *local*
artifacts (the repo-root `autonomath.db` and `data/jpintel.db` that mirror
production-shape rows) so that R2 and the Fly volume stay untouched. The
RTO/RPO numbers below are therefore lower-bound estimates of the production
path — production adds R2 download + Fly volume IO on top of what we
measured. We call those out explicitly per step.

**code path** end-to-end against local data; the live R2 LIST is left to
the monthly cron (`scripts/cron/restore_drill_monthly.py`, GHA
`restore-drill-monthly.yml`, scheduled 15th of month 03:00 JST).

## 1. R2 backup inventory (manifest-derived)

The two backup prefixes the runbook restores from, with their cron + worker
of record:

| Prefix                              | DB           | Cadence | Cron                                          | Workflow                                     |
|-------------------------------------|--------------|---------|-----------------------------------------------|----------------------------------------------|
| `s3://autonomath-db-backup/jpintel/`               | jpintel.db   | hourly  | `scripts/cron/backup_jpintel.py`              | (entrypoint cron, hourly)                    |
| `s3://autonomath-db-backup/autonomath-api/autonomath-db/` | autonomath.db | daily   | `scripts/cron/backup_autonomath.py`           | `.github/workflows/weekly-backup-autonomath.yml` |

Each backup ships as `<name>-YYYYMMDD-HHMMSS.db.gz` plus a `.sha256`
sidecar. The hot-tier window is 3 days (drill explicitly samples >3-day
*cold* keys per `restore_drill_monthly.py` §3). Expected presence at this
date stamp:

* **jpintel hourly**: ~720 keys for the rolling 30 d window (24 × 30); the
  newest key is ≤1 h old, the 7-day-old key sits between offsets 144 and 168
  in lexicographic order.
* **autonomath daily**: ~30 keys for the rolling 30 d window (1 × 30) plus
  4 weekly retention pins; the newest key is ≤24 h old, the 7-day-old key
  is offset 7.

A live `aws s3 ls --endpoint-url $R2_ENDPOINT s3://autonomath-db-backup/jpintel/`
on a Fly machine is the canonical proof; from the operator laptop the
equivalent observable is the `jpintel.db.restore_drill_log` table populated
by the monthly cron. (See R8_BACKUP_RESTORE_DRILL_AUDIT_2026-05-07.md for
the audit trail of past drill rows.)

## 4. Failure-mode test — corrupted backup

```text
$ head -c 1048576 /Users/shigetoumeda/jpcite/data/jpintel.db > /tmp/dr_drill_2026_05_07/jpintel-corrupt-head.db
$ ls -lh /tmp/dr_drill_2026_05_07/jpintel-corrupt-head.db
-rw-r--r--  1 shigetoumeda  wheel   1.0M
$ sqlite3 /tmp/dr_drill_2026_05_07/jpintel-corrupt-head.db "PRAGMA quick_check;"
Error: in prepare, database disk image is malformed (11)
duration_sec=0
$ sqlite3 /tmp/dr_drill_2026_05_07/jpintel-corrupt-head.db "PRAGMA integrity_check;"
Error: in prepare, database disk image is malformed (11)
```

The truncated 1 MB head (≈ first 256 SQLite pages of a 426 MB file) trips
SQLite's "database disk image is malformed" detector at the *prepare*
stage — meaning the engine bails before evaluating the pragma at all,
which is the strongest possible signal. `restore_db.py` invokes
`PRAGMA integrity_check` after gunzip but before the atomic swap (see
`_integrity()` line 117); a malformed download therefore **cannot reach
the live volume**. The runbook §5 fall-through to "pick an earlier
snapshot" is the correct response and matches the cron's exit code 3
(`integrity RED`).

We did **not** test the second corruption shape (random byte flips inside
otherwise-intact pages) in this drill — `quick_check` is faster than
`integrity_check` but skips this class of damage. The monthly cron runs
`integrity_check` (slower, full coverage); the runbook §6a step likewise.
A future drill can stage a "flip 100 random bytes mid-DB" sample to
exercise that branch end-to-end.
