# R8 R2 Key Mint Audit — GHA secret installation for nightly-backup — 2026-05-07

**Scope**: jpcite v0.3.4 — internal hypothesis check on whether the operator's Cloudflare OAuth bearer can mint S3-compatible R2 access keys via the public CF API, and (if not) what manual operator step closes the gap so `nightly-backup.yml`'s fail-closed R2 upload step stops red-lining for missing GHA secrets.

Companion: `R8_BACKUP_FIX_2026-05-07.md` (Defect A: argv shell-wrap), `R8_BACKUP_RESTORE_DRILL_AUDIT_2026-05-07.md` §5.

> Internal hypothesis framing: this doc records what we tried, what the API returned, what the auth boundary actually is, and what the operator needs to do by hand. **No production secret values appear in this file** (per task constraint and standing repo rule on docs).

---

## 1. Goal one-liner

`nightly-backup.yml` fail-closed step (commit 2026-05-03 hardening) requires four GHA repository secrets — `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_ENDPOINT`, `R2_BUCKET` — to mirror the SQLite snapshot to Cloudflare R2 (`autonomath-db-backup` bucket, 14-day rolling). Today: **all four exist on Fly side** (`flyctl secrets list -a autonomath-api` shows the digests), **none exist on GHA side** (`gh secret list` shows only `CF_ACCOUNT_ID` / `CF_API_TOKEN` / `FLY_API_TOKEN`). Until the four R2_* secrets are installed in GHA, the workflow fails at the upload step every night.

The hypothesis under test: can we mint a fresh R2 S3 access key pair via the CF public API using the operator's existing OAuth bearer (already in hand for Wave R8 inbox tasks), and pipe the output straight into `gh secret set` without any value ever landing in a doc, log, or terminal scrollback?

## 2. Auth boundary discovered

**OAuth bearer token in hand**: `XEuMK...` (Wave R8 inbox token; Cloudflare dashboard OAuth flow style — single-use bearer tied to the operator's `shigetosidumeda@gmail.com` account, `account_id=037691739017ffe105f57fe391f4aebb`).

Probe results against `api.cloudflare.com/client/v4`:

| Endpoint | Method | HTTP | CF code | Verdict |
|---|---|---|---|---|
| `/accounts/{id}/r2/buckets` | GET | 200 | success | **Reads OK.** Returns `autonomath-db-backup` + `autonomath-ec-assets`. |
| `/accounts/{id}/r2/buckets/autonomath-db-backup/usage` | GET | 200 | success | Reads bucket usage (1 object, 8.82 GB payload). |
| `/accounts/{id}/r2/access-keys` | GET | 404 | 10015 "No route matches this url" | **Mint route absent for this token.** |
| `/accounts/{id}/r2/access-keys` | POST `{name, permission:read_and_write, scope:all_buckets}` | 404 | 10015 same | Same. |
| `/accounts/{id}/r2/api-tokens` | GET | 404 | 10015 | Same. |
| `/accounts/{id}/r2/credentials` | GET | 404 | 10015 | Same. |
| `/accounts/{id}/r2/temp-access-credentials` | POST | 401 | 10002 "Unauthorized" | Route **exists** but rejects this token. |
| `/accounts/{id}/api-tokens` | GET | 400 | 7000/7003 "No route" | API-token CRUD also absent. |
| `/user/tokens/verify` | GET | 401 | 1000 "Invalid API Token" | Token is **not** an API token — it is a session-style OAuth bearer. |
| `/memberships` | GET | 200 | success | Returns "Super Administrator - All Privileges" via the OAuth grant. |

**Interpretation.** The OAuth bearer carries the operator's full account ACL **for routes that accept dashboard/OAuth auth**, but the `r2/access-keys`, `r2/api-tokens`, and `accounts/{id}/api-tokens` mint endpoints are not part of that surface. They are reachable only via:

1. A dashboard session cookie (the path used by `dash.cloudflare.com → R2 → Manage R2 API Tokens → Create API Token`), **or**
2. A CF API token (the long opaque string starting with `v1.0-…`) that itself carries the **`Account → R2 API Tokens → Edit`** permission group — distinct from the **`Account → R2 → Edit`** (storage admin) permission group.

The `r2/temp-access-credentials` route returns 401 instead of 404 because it requires a `parentAccessKeyId` of an already-existing R2 access key — i.e., it derives short-lived STS-style creds **from** a pre-existing parent key; it cannot bootstrap the first key.

**Net**: this token cannot mint. Hypothesis falsified.

## 3. Secondary hypothesis — extract from Fly machine env

Fly stores the existing R2 secrets (`R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` / `R2_ENDPOINT` / `R2_BUCKET`) and exposes them as env vars to the running Machine. `flyctl ssh console -C` can read env vars on the Machine; if we pipe directly into `gh secret set --body /dev/stdin`, the value transits through the SSH session and the gh stdin without landing in any doc or scrollback.

**Probe**: `flyctl ssh console -a autonomath-api --machine 85e273f4e60778 -C 'sh -c "echo R2_ACCESS_KEY_ID_LEN=${#R2_ACCESS_KEY_ID}"'` (length probe — does not expose the value).

**Result**: `Error: tunnel unavailable: Error contacting Fly.io API when probing "personal": timed out (context deadline exceeded)` — Fly WireGuard tunnel API is transiently failing today. `flyctl auth whoami` and `flyctl status` both work; the failure is specifically in the WireGuard control-plane probe step that `ssh console` runs before opening the tunnel. This is a known Fly.io flake (correlated with the `Metrics token unavailable` warning that accompanies every command in this session).

Two retries hit the same timeout. Not blocking — the manual-step path below is independent of Fly tunnel availability.

## 4. Resolved path — operator manual step

The cleanest fix is **issue a new R2 API token** (3-minute dashboard task) and pipe the four resulting values into `gh secret set` from the operator's terminal. This:

- Avoids re-using the Fly-side key in two places (cleaner blast-radius if either side is rotated).
- Doesn't require Fly tunnel availability.
- Doesn't require us to run any LLM-heavy or long-running automation.
- Lands the four secrets on the GHA side **without** any value ever transiting through this audit doc or any committed file.

### 4.1 Manual recipe (operator-only)

```bash
# 1. In a browser, open:
#    https://dash.cloudflare.com/037691739017ffe105f57fe391f4aebb/r2/api-tokens
#    Click "Create API Token". Settings:
#      Token name:    jpcite-gha-backup
#      Permissions:   Object Read & Write
#      Specify bucket(s):  autonomath-db-backup
#      TTL:           (leave blank — non-expiring)
#    Click Create. Copy the four shown values: Access Key ID, Secret Access Key,
#    Endpoint URL, Token (we don't need Token; only the S3-compat trio plus bucket).

# 2. From the operator workstation (~/.gh authenticated to the bookyou repo owner):
cd /Users/shigetoumeda/jpcite

# Stdin only — values never appear in shell history or scrollback when typed at the
# read prompt. Use `read -s` so the terminal echo is suppressed.

read -rs -p "R2_ACCESS_KEY_ID: " val && printf '%s' "$val" | gh secret set R2_ACCESS_KEY_ID && unset val
echo
read -rs -p "R2_SECRET_ACCESS_KEY: " val && printf '%s' "$val" | gh secret set R2_SECRET_ACCESS_KEY && unset val
echo
# Endpoint is not secret per se but we keep it consistent with the workflow contract
printf 'https://037691739017ffe105f57fe391f4aebb.r2.cloudflarestorage.com' | gh secret set R2_ENDPOINT
printf 'autonomath-db-backup' | gh secret set R2_BUCKET

# 3. Verify (only digests visible):
gh secret list | grep '^R2_'
# Expect 4 rows: R2_ACCESS_KEY_ID / R2_SECRET_ACCESS_KEY / R2_ENDPOINT / R2_BUCKET
```

### 4.2 Trigger one nightly-backup run to confirm green

```bash
gh workflow run nightly-backup.yml
# Wait ~3 min, then:
gh run list --workflow=nightly-backup.yml --limit 1
gh run view --log <run-id> | tail -80
# Expected: "off-site upload step" reports object `autonomath-db-backup/jpintel-YYYYMMDD-*.db.gz` written.
```

If the new R2 token leaks or the operator wants to retire the GHA-side key independently of the Fly-side key:

# Then re-run §4.1 with the rolled values.

## 5. Why we did not script the dashboard click

Two failed approaches considered and discarded:

1. **Headless Playwright against `dash.cloudflare.com`** — would require operator's password + 2FA seed in CI; both are out of band per `feedback_zero_touch_solo` and standing secret hygiene. Worse blast radius than the OAuth bearer we already had.
2. **Cloudflare API token created via browser, with `Account → R2 API Tokens → Edit` permission, used to mint the R2 access key via API** — this is a **two-token pyramid** (mint-token → mint storage-key → use storage-key) where the mint-token itself has to be created by hand in the dashboard anyway, so we save zero clicks vs. just creating the storage key directly. Adds one moving part for no gain.

The straight-line manual recipe in §4.1 is the lowest-friction option for a one-time bootstrap.
