## よく聞かれる質問: 「AI で採択確率、出してくれない？」

AutonoMath (9,998 件の補助金データを返す API / MCP サーバー) を公開してから、よく聞かれるのがこれです。

> 「うちの会社で A 補助金出したら採択される確率、AI で出せないですか？」
> 「過去の採択事例から機械学習で 『採択率 73%』みたいな数字出せますよね？」

技術的には出せます。sklearn で書けば半日で動くモデルができる。でも AutonoMath は**これを実装していないし、今後もする予定がありません**。ヨイショではなく、構造的に詐欺になるからです。

## 採択率予測が詐欺スレスレになる 4 つの構造的理由

### (1) 公募ごとに審査基準が非公開かつ変動する

補助金の審査基準は公募要領に「事業性」「実現可能性」「波及効果」のような**大項目しか書かれていない**。採点ルーブリックは審査員の手元にあり、年度で変わる。さらに審査員自体が毎年入れ替わる。

機械学習の前提は「訓練時と推論時の分布が同じ」。分布が年度ごとに変わる (データドリフトどころかコンセプトドリフト) 対象に回帰モデルを当てるのは、予測とは呼ばない。

### (2) 母数が小さい

主要な補助金の採択件数は年 30-200 件。そこから「採択:不採択 = 1:4」程度の比率で訓練データを作ると、実質 150-1000 サンプル。特徴量を 20 個作った時点で次元の呪いの手前です。

ROC-AUC 0.7 が出たとしても、それは過学習か、あるいは「大企業ほど通りやすい」みたいな当たり前のシグナルを拾っているだけです。

### (3) 機械学習で回帰する → 因果ではなく相関

仮に「過去データで ROC-AUC 0.85」が出たとしても、それは**過去の採択者と似た属性の人が採択される傾向**を表しているだけで、「この特徴量をこう変えれば採択される」という因果ではない。

ユーザーは必ず「確率を上げるには何をすれば？」と聞いてくる。これに回帰モデルの partial dependence plot で答えるのは、因果推論を放棄した回答です。

### (4) ユーザーは「◯ %」という数字を信じてしまう

ここが一番深刻。人間は「採択確率 73%」と言われると、その数字の**不確実性を捨てて**意思決定する。

- 実装者は 「95% 信頼区間 23%-91%」と内部では分かっている
- でも UI に出すのは「73%」だけ
- ユーザーは 73% を 73% として受け取り、200 万円の申請準備コストを投下する
- 落ちる。「73% って言ったじゃないですか」

これはキャリブレーションの問題で、技術的には確率校正すれば改善できる、ということにはなっている。しかしそれでも (1)(2)(3) が解決しないので、根本的に出せない数字です。

## 補助金適正化法 29 条との関係

日本には「補助金等に係る予算の執行の適正化に関する法律」があります。29 条は虚偽申請の罰則規定で、**5 年以下の懲役または 100 万円以下の罰金** (あるいは併科) です。

:::message alert
「AI が出した採択予測 73% だったから行けると思った」は免責になりません。申請者本人の責任です。API 提供者として、ユーザーに刑事リスクを背負わせる数字を出すことはできない。
:::

さらに、AI による採択率予測を根拠に「絶対通ります」と営業をかける代理業者が実在していて、これは特商法・景品表示法違反で摘発事例もあります。API 事業者が予測を提供すると、この種の代理業者に武器を渡すことになる。

## 代わりに提供しているもの

AutonoMath が予測のかわりに返しているのは3種類です。

### (a) 採択事例 (生データ)

過去の採択者の情報で公開されているもの (法人名、金額、事業概要) を、制度ごとに紐付けて返す。ユーザーは自分で「自社と近い事例」を判断する。

### (b) 除外ルール (機械的に判定できるもの)

「世帯所得 1000 万円以下」「従業員 300 人以下」「法人設立後 10 年以内」のような**数値で機械判定できる要件**を 35 本の除外ルールとして提供。これは予測ではなく、申請資格があるかどうかの事実判定です。

```python
# 例: check_exclusions tool の返り値
{
  "unified_id": "saitama_subsidy_maff_xxx",
  "excluded": true,
  "reason": "household_income > 10,000,000 (rule: MAFF_INCOME_CAP_2025)",
  "source_url": "https://www.maff.go.jp/..."
}
```

「確率」ではなく「該当/非該当」を返す。ここは詐欺にならない。

### (c) 一次資料 URL

全制度に `source_url` が必ず入っている。ユーザーと LLM が最終的に原文を読めるようにする。

## データを出す。予測は出さない。

API として誠実な線引きはここだと思っています。

- **事実** (制度名、金額上限、期限、除外要件) → 返す
- **判定** (数値で機械判定できる範囲) → 返す
- **予測** (採択確率、スコアリング、ランキング) → 返さない

「採択率予測」を実装したら売れるのは分かっています。でも売れるから作る、が許される領域ではない。5 年失敗したら申請者は廃業します。
