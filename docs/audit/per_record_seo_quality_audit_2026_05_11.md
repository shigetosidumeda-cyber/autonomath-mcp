## Top-5 即修正

1. **Cases extensionless fix** — `scripts/generate_case_pages.py` line 83 `case_url()`:
   change `f"https://{DOMAIN}/cases/{case_id}.html"` → `f"https://{DOMAIN}/cases/{case_id}"`.
   Filenames on disk keep `.html`, only canonical / og:url / sitemap loc strip the extension.
   Add `_redirects` rule `/cases/:slug.html /cases/:slug 301`. Affects 2,286 pages + 2,286 sitemap rows.
2. **`sitemap-index.xml` inclusion** — `scripts/sitemap_gen.py` (generator for the index) is missing
   `sitemap-cases.xml`, `sitemap-laws.xml`, `sitemap-laws-en.xml`, `sitemap-enforcement-cases.xml`
   from its child-sitemap list. Google currently sees 0 of the 9,964 new URLs.
   Add 4 `<sitemap>` entries pointing at those files. (Existing `sitemap-enforcement.xml` covers
   only the legacy 300 act-* family.) **This is the single highest-leverage fix in the audit.**
3. **Cases BreadcrumbList JSON-LD** — `scripts/generate_case_pages.py` `build_jsonld()` returns only
   `Article`. Add a second `<script type="application/ld+json">` with `BreadcrumbList`
   `{ホーム→採択事例→pref?→company}`. Already done in laws + enforcement; copy that pattern.
4. **Root index pages** — `site/cases/index.html` and `site/laws/index.html` do NOT exist; the
   visible breadcrumb on every per-record page links to a 404. `site/enforcement/index.html` exists
   (already covers 22,010 total enforcement records). Generate two analogous index pages:
   - `/cases/index.html` grouped by prefecture (47 buckets + "全国") with sitelinks to ~5
     representative case_studies per bucket.
   - `/laws/index.html` grouped by `am_law.category` (公布 / 改正 / 政省令 / 通達 etc) with first
     50 laws per bucket + a-z navigation. Cap rendered list at ~500 to keep page <1 MB.
5. **Cross-record JSON-LD `mentions` array** — cases / laws / enforcement all silently drop
   machine links to related records in their JSON-LD even when the HTML carries them. Wire:
   - cases.Article.mentions = laws cited via `programs_used_json → program_law_refs → laws.slug`
   - laws.Legislation.citation = court_decisions / enforcement_cases that reference this lawid
   - enforcement.GovernmentService.about = the houjin entity (`@id: /entities/houjin/{bangou}`)
   Each link adds `@type: Thing` + `@id: https://jpcite.com/{...}` so Schema validators see graphs.

## 残 ~45 chunk push 戦略

Decision: **regenerate, then push the full 9,964 corpus from a clean run** rather than diff-patching
the already-pushed 1,200. Reasoning:

- Fixes #1 / #3 / #5 above touch every cases page (2,286) and every laws/enforcement JSON-LD block.
  Patching in-place over SFTP is more expensive than re-emitting from `data/jpintel.db` +
  `autonomath.db`, which takes 4-7 min total per generator (clocked Wave 11).
- The Wave 12 push delta was the bottleneck (1,200 in chunk 0–5), not generation. The new chunk
  plan ships the same surface twice — once now (fix #1 + #3 only, ~9.4 MB total HTML re-upload),
  then a second pass after #4 + #5 land. Splitting the cost across two pushes keeps each chunk
  under the 250-MB Cloudflare wrangler ceiling that bit Wave 12.
- Idempotent write paths already exist in laws (`existing != ja_html` check, line 1115) and
  enforcement (line 678). cases lacks one — adding `if existing == page: continue` to the cases
  generator (around line 571) drops 80% of the rerun cost when only metadata changes.
- Chunk strategy: 50 chunks × ~200 pages each, `scripts/ops/seo_chunk_push.py` SFTP per chunk,
  retry on transient TLS / wrangler 524. Estimated 90-120 min wallclock for the next session,
  zero LLM cost.

Push order (highest SEO impact first): 1) cases (after fix #1) → 2) `sitemap-index.xml` (fix #2,
single file, <2 KB upload) → 3) `/cases/index.html` + `/laws/index.html` (fix #4) → 4) laws batch →
5) enforcement batch → 6) `sitemap-llms.xml` aggregator (new file, see below).
