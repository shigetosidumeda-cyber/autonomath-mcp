# Audit Trail — Work-Paper Reproducibility for 会計士

## Why this exists

Auditors (会計士 / 税理士 / 内部監査担当) cite our dataset in their work-papers
when evaluating client compliance. The cite needs three properties:

1. **Identity** — a single string that names the corpus state at the moment
   of evaluation. "Latest as of 2026-04-29" is not enough — the cron may have
   ingested ten amendments on 2026-04-29 between 03:00 and 23:00.
2. **Reproducibility** — a year later, calling the same endpoint with the
   same parameters must either return byte-identical results, or return a
   *different* identity, proving the corpus mutated.
3. **Tamper-evidence** — auditors need to detect mutation even if the
   timestamp happens to land on the exact same minute (vanishingly rare,
   but legally relevant — work-papers must withstand cross-examination).

Per-row `fetched_at` is too granular for any of these. It tells you when
each individual row was scraped — not "the corpus, viewed as a whole, was in
state X at time T". This document defines the fields that satisfy 1-3 and
how to consume them.

---

## The two fields

Audit-relevant endpoints (see "Coverage" below) inject two top-level keys
onto every successful response:

```jsonc
{
  "corpus_snapshot_id": "2026-04-29T03:14:07Z",
  "corpus_checksum":    "sha256:9f8e7d6c5b4a3210",
  // ... endpoint-specific payload ...
}
```

### `corpus_snapshot_id`

ISO-8601 UTC timestamp. Computation order:

1. `MAX(detected_at)` from the 制度改正履歴 — the cron's amendment detector
   records every observed mutation. This is the **strongest** signal:
   monotonically increasing, generated by our own データ取込, never falsified.
2. If the 制度改正履歴 is unavailable (e.g. jpintel.db connection without
   cross-DB ATTACH), fall back to
   `MAX(fetched_at)` across canonical corpus tables (`programs`, `laws`,
   `tax_rulesets`, `court_decisions`).
3. If no rows exist, the value is `1970-01-01T00:00:00Z` (deterministic
   placeholder so the field is always present).

### `corpus_checksum`

`sha256:<16hex>` deterministic over the inputs:

```
sha256( snapshot_id || api_version || row_count_csv )
```

where `row_count_csv` is `COUNT(*)` from the four canonical tables in a
fixed order. The hash is truncated to 16 hex chars (64 bits of entropy —
enough to make accidental collision negligible in practice; cryptographic
authentication is not the goal, change-detection is).

Mutation in **any** canonical table flips at least the row-count input and
thus the checksum, even when `snapshot_id` happens to be unchanged
(e.g. multiple amendments detected in the same second).

## How to use it in a work-paper

A standard work-paper entry looks like:

> 評価日時: 2026-04-29 12:34 JST
> 評価対象: 当社 (資本金 30,000,000 円, 青色申告) の 試験研究費税額控除 (一般型) 適用可否
> 使用 API: jpcite `/v1/tax_rulesets/evaluate`
> リクエスト: `{ "business_profile": {...}, "target_ruleset_ids": ["TAX-ca4b993ca5"] }`
> レスポンス: `applicable=true`, reasons=["blue_form_return_filed == True", "research_expense_jpy >= 1"]
> **Corpus snapshot:** `corpus_snapshot_id=2026-04-29T03:14:07Z`,
>                    `corpus_checksum=sha256:9f8e7d6c5b4a3210`
> 監査員署名: \_\_\_\_\_\_\_\_

### Reproducibility check (one year later)

To verify the evaluation, the auditor re-runs the same request. Three cases:

| Outcome                               | Interpretation                          |
|---------------------------------------|------------------------------------------|
| Same `applicable` + same checksum      | Corpus unchanged. Result still cite-able. |
| Same `applicable` + different checksum | Corpus mutated but verdict survives — note the new checksum, evaluation still defensible. |
| Different `applicable`                 | Corpus mutated AND verdict changed. Document the date of change (use `corpus_snapshot_id` deltas) and re-evaluate. |

This is a HONEST reproducibility check. We do **not** claim
cryptographic non-repudiation — the checksum is a 64-bit truncation, the
corpus snapshot timestamp is from our own clock. But for a work-paper that
needs to survive cross-examination, "the jpcite corpus checksum
flipped between my two runs, so I documented the new state on $date" is a
defensible audit trail.

## Coverage

The following endpoints inject `corpus_snapshot_id` + `corpus_checksum`
on every 200 response:

| Endpoint                                      | Notes |
|-----------------------------------------------|-------|
| `POST /v1/tax_rulesets/evaluate`              | Primary use-case. Auditor evaluates client profile against multiple rulesets, work-paper cites the verdict + the snapshot pair. |
| `GET  /v1/tax_rulesets/{unified_id}`          | Single-ruleset lookup. Snapshot pair lets the auditor re-check the rule body itself for amendments. |
| `GET  /v1/programs/{unified_id}`              | Program citation in adoption / clawback work. |
| `GET  /v1/laws/{unified_id}`                  | 法令本文 evidence. e-Gov updates land via cron — checksum flips when the law text changes. |
| `GET  /v1/court-decisions/{unified_id}`       | 判例 evidence. 課税庁の処分取消訴訟 work-paper for tax-position disputes. |

Search endpoints (`*/search`) do NOT inject these fields by default — the
search payload is a paginated discovery surface, not a citation target.
Auditors who need a snapshot-pinned search should follow up each
`unified_id` with the corresponding get-by-id call.

---

## Caching semantics

The snapshot computation is cached process-locally for **5 minutes** keyed
by the connection's database file path. A process serving 1k req/sec
recomputes the snapshot ~12 times/hour, not 3.6M.

Auditors who need second-by-second precision (e.g. evaluating around a
known cliff date such as 2026-09-30 / 2027-09-30 / 2029-09-30) can pass the
`X-jpcite-Snapshot-Refresh: 1` header to bypass the cache. (Header
support is opt-in and reserved for licensed audit tools — see
`/v1/me/audit-tools` for activation.)

## Failure modes

| Failure                                 | Endpoint behavior                            |
|-----------------------------------------|-----------------------------------------------|
| 制度改正履歴 absent                     | Falls back to `MAX(fetched_at)` (silent).    |
| All canonical tables empty              | `corpus_snapshot_id="1970-01-01T00:00:00Z"`, checksum still computed. |
| sqlite3 connection raises during compute | Snapshot helper does not raise — the per-request logging records a 500 root cause but the response still ships placeholder values starting with `unknown-`. |

This is intentional: an auditor who sees `1970-01-01` or `unknown-` in a
work-paper response should treat it as a load error and re-fetch, not as a
silent zero-value.
