## 1. Why H4

H3 collapsed the agent entry SOT into `AGENTS.md`. The next layer that needed
shrinking was the CI gate taxonomy itself:

1. `.github/workflows/` had grown to **168 YAML files**, but
   `.github/workflows/README.md` still claimed **82 files / v0.3.4**. The
   README headline and the live file set drifted by 86 files.
2. `scripts/ops/release_readiness.py` reported **9/9 PASS** on a SHA where
   `scripts/ops/sync_workflow_targets_verify.py --check` simultaneously
   reported **120 drift rows**. Two gates, two different verdicts on the
   same tree.
3. `make mcp` only ran the static manifest drift checker
   (`scripts/check_distribution_manifest_drift.py`) — a developer who ran
   `make mcp` and saw `PASS` had no way to learn that runtime FastMCP tool
   count was off, or that public manifests (`site/mcp-server.json` etc.)
   were stale.
4. Two distinct MCP drift checkers live side-by-side
   (`scripts/check_distribution_manifest_drift.py` static vs
   `scripts/check_mcp_drift.py` runtime+range) with no doc explaining when
   to use which.

H4 closes all four gaps in one landing.

## 2. The 5-category taxonomy

Every workflow lands in exactly **one** of five categories. No
double-counting.

| Category | Count | Trigger contract | Failure semantics |
|---|---:|---|---|
| PR hard gate | 11 | `pull_request:` on `main` | Blocks merge |
| Release hard gate | 5 | `push: tags: v*` + `push: branches: main` + `pull_request:` | Blocks tag → publish |
| Deploy gate | 5 | `workflow_run:` on green test or `push: branches: main` | Aborts deploy mid-flight |
| Scheduled diagnostic | 138 | `schedule:` cron | Alert-only; Slack/Telegram/email page |
| Manual operator-only | 9 (+6 v3 probes) | `workflow_dispatch:` ONLY | Operator must explicitly fire |

**Total: 168** (= `ls .github/workflows/*.yml \| wc -l` on 2026-05-17).

Workflow-by-workflow assignment lives in
[`.github/workflows/README.md`](../../.github/workflows/README.md) §1..§5
(regenerated by this landing).

### Edge cases reconciled

- `lane-policy-warn.yml` is PR-triggered but `rc=0` always. Classified as
  PR hard gate because the PR-comment surface participates in merge
  gating, even though it cannot fail the check.
- `dependabot-auto-merge.yml` only fires on dependabot PRs but participates
  in branch-protection contract. Classified as PR hard gate.
- `codeql.yml` and `tls-check.yml` are scheduled but also gate release.
  Classified as release hard gate because their failure must block a `v*`
  tag from shipping.
- `narrative-sla-breach-hourly.yml` and `same-day-push-cron.yml` have
  `push:` triggers in addition to cron. Classified as scheduled
  diagnostic — the `push:` is for hot-reload of the cron logic, not a
  gate.
- The 6 `*_v3` drift probes (`mcp_drift_v3`, `facts_registry_drift_v3`,
  `fence_count_drift_v3`, `openapi_drift_v3`, `sitemap_freshness_v3`,
  `structured_data_v3`) are `workflow_dispatch:`-only. Classified as
  manual operator-only.

## 3. Makefile target rename + split

Before:

```make
mcp:
    $(PY) scripts/check_distribution_manifest_drift.py
```

After:

```make
mcp: mcp-static    # backwards-compat alias

mcp-static:
    $(PY) scripts/check_distribution_manifest_drift.py

mcp-runtime:
    $(PY) scripts/check_mcp_drift.py

mcp-public-discovery:
    $(PY) scripts/sync_mcp_public_manifests.py
    $(PY) scripts/check_distribution_manifest_drift.py
    $(PY) scripts/check_sitemap_freshness.py
```

| Target | Boots MCP? | Net effect | Pairs with CI gate |
|---|---|---|---|
| `make mcp-static` | no | Static cross-check of pyproject / server.json / dxt / smithery / mcp-server.json | `distribution-manifest-check.yml` (§1) |
| `make mcp-runtime` | yes | Asserts `len(await mcp.list_tools())` matches floor + range | (gate substrate — not yet a CI workflow) |
| `make mcp-public-discovery` | no | Regenerates public manifests + static drift + sitemap freshness | `mcp-registry-publish.yml` + `sitemap_freshness_v3.yml` (§5) |

`make mcp` keeps the legacy semantics (alias to `mcp-static`) so that
`make e2e` (`bootstrap → api → mcp → sync-manifest-sha → validate → gate`)
continues to pass without modification.

## 4. release_readiness.py — 9 → 10 checks

Added `check_workflow_targets_full_drift(repo_root)` (file
`scripts/ops/release_readiness.py`).

Behaviour:

1. Probe for `scripts/ops/sync_workflow_targets_verify.py`. If absent
   (synthetic seed-repo case in `tests/test_release_readiness.py`),
   degrade to PASS — the existing
   `check_workflow_targets_git_tracked` already covers the
   forward-direction drift class.
2. If present, shell out to `python3 scripts/ops/sync_workflow_targets_verify.py --check`.
   Exit 0 → PASS. Non-zero → FAIL with the last 3 stdout lines as
   evidence.
3. Read-only by contract — the verifier is stdlib + `git ls-files` only,
   no network, no mutation.

Why a 10th check rather than tightening the 9th: the 9th check
(`workflow_targets_git_tracked`) only verifies that **declared** targets
exist in git. It cannot detect **new** test files that should be wired
into the workflow env lists. That inverse drift is precisely what the
verify script catches (and what `check-workflow-target-sync.yml`
reports as PR-side `[FAIL] 120 drift row(s) detected.`).

Test coverage:

- `tests/test_release_readiness.py::test_workflow_targets_full_drift_skips_when_verifier_missing`
  (new — verifies graceful degrade).
- `tests/test_release_readiness.py::test_build_report_passes_for_release_ready_repo`
  (existing — total bumped 9 → 10).

Verification on the live tree:

```
$ .venv/bin/python scripts/ops/release_readiness.py | jq .summary
{ "pass": 9, "fail": 1, "total": 10 }
```

The 1 fail is the new `workflow_targets_full_drift` check correctly
surfacing the 120-row drift that motivated H4. Operator can clear it
with `python scripts/ops/sync_workflow_targets.py --apply` (DEEP-49) and
commit the env updates.

## 5. Anti-patterns the 2-checker design avoided

- **Do not collapse the two MCP drift checkers** into one. They have
  distinct semantics:
  - `check_distribution_manifest_drift.py` is **static**, fast, hermetic
    — runs in `<1s`, no network, no FastMCP boot. Suitable for every
    push + PR.
  - `check_mcp_drift.py` is **runtime** — actually imports
    `jpintel_mcp.mcp.server`, awaits `list_tools()`, and asserts the
    count + range bands. Necessary for catching stale-but-valid
    manifests, but cannot run on every PR (boot cost + DB dependency).
- **Do not auto-fix the workflow target drift inside
  `release_readiness.py`**. The verifier ships only in `--check`
  mode. Auto-apply lives in
  `scripts/ops/sync_workflow_targets.py --apply` so that human review
  remains in the loop on which new tests join the gating set.
- **Do not classify warn-only PR comments as scheduled diagnostics**.
  `lane-policy-warn` rc=0 by design, but its purpose is PR-side
  signalling, not background observability. Misclassifying would let
  the next operator silence it as "alert only".

## 8. What NOT to do downstream

- Do not "fix" the new check failing on the live tree by skipping it.
  The failure is the **signal** — run `sync_workflow_targets.py --apply`
  and commit the env updates instead.
- Do not delete the `make mcp` alias. It is consumed by `make e2e`.
- Do not extend `release_readiness.py` to run network probes. The
  contract is "local files only; no network; no mutation". Network
  probes belong in the cron diagnostics (§4).
- Do not promote `make mcp-runtime` into a CI gate without first
  building a hermetic boot fixture — the runtime check requires the
  `autonomath.db` to be present, which is a no-go on a 9.7 GB volume
  on GHA runners (see `feedback_no_quick_check_on_huge_sqlite`).
