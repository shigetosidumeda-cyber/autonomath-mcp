## Goal
50% wall-time reduction on `ci.yml` (= `test.yml` in this repo) + `release.yml`.

## Levers applied

### 1. `actions/cache@v4.2.0` for ruff / mypy / pytest

Pinned by SHA `1bd1e32a3bdc45362d1e726936510720a7c30a57`. Keys bust on `pyproject.toml` hash so dependency bumps invalidate cleanly. `restore-keys` falls back to a per-OS partial match so even cold-key runs get _some_ acceleration.

- `.ruff_cache`: warm hit drops `ruff check` + `ruff format --check` on ~100 files from ~10-15s to ~2-3s.
- `.mypy_cache`: warm hit drops `mypy src/` (the heaviest tail step) from ~3-4 min cold тЖТ ~30-60s warm.
- `.pytest_cache`: lighter savings (collection metadata + last-failed bookkeeping); ~5-10s per shard.

`mypy_cache` key includes `hashFiles('src/**/*.py')` so incremental mode runs on a fresh-enough cache; large refactors invalidate cleanly.

### 2. Matrix sharding (release.yml)

`release.yml::pytest` previously ran all ~500 test files on one runner. Now it shards 4-way (same modulo as `test.yml`):

```yaml
strategy:
  fail-fast: false
  matrix:
    shard-index: [0, 1, 2, 3]
    shard-count: [4]
```

The shard-builder script is identical to `test.yml`'s; deterministic + index-modulo so the 4 shards are stable and re-runnable.

Same `PYTEST_TARGETS` env list is reused in both workflows (single source of truth).

mypy stays single (`test.yml::mypy`) тАФ cache makes it fast enough that sharding mypy would add startup overhead without speedup.

### 3. `needs:` / dependency restructure (test.yml)

`test.yml` now has 5 independent parallel jobs (no `needs:` between them):

- `pytest` (12 shards) тАФ pure pytest, no lint/typecheck/docs side-quests
- `ruff` (lint + format) тАФ sub-30s on warm cache, ~1 min cold (single Py3.12 runner; ruff-only install, no `pip install -e .[dev,site]`)
- `mypy` (`--continue-on-error: true`) тАФ best-effort gate per CLAUDE.md, separated so failure doesn't block green pytest
- `pip-audit` (`continue-on-error: true`) тАФ CVE supply chain scan
- `mkdocs` тАФ `mkdocs build --strict`, only `docs-requirements.txt` installed

Wall time of the workflow = `max(pytest_slowest_shard, mypy, ruff, mkdocs, pip-audit)` not `pytest_slowest + mypy + ruff + mkdocs + pip-audit`.

`release.yml` was already `test тЖТ build тЖТ publish-pypi тЖТ github-release` and that chain is **correct** (PyPI publish must wait on green tests + build artifact). The change is that `test` is now split into `ruff || pytest` (parallel) and `build` declares `needs: [ruff, pytest]`.

### 4. `fail-fast: false`

Already set on `test.yml::pytest`. Added to `release.yml::pytest` (new shard matrix). Means a green-light shard surfaces alongside red shards rather than being cancelled тАФ surfaces ALL failures in one run for faster triage than serial re-runs.

## Estimated wall time (after)

### test.yml

- `pytest` longest shard: ~10-12 min (unchanged; pytest is the floor)
- `ruff`: ~1 min cold, ~30s warm
- `mypy`: ~3 min cold, ~45-60s warm (was inline ~3-5 min serial behind pytest shard 0)
- `pip-audit`: ~1 min
- `mkdocs`: ~1-2 min

**Total wall time = ~10-12 min** (driven by slowest pytest shard).

vs. before **~15-18 min** (pytest + serial mypy+ruff+mkdocs tail on shard 0).

**Reduction: ~35-45%** on test.yml. The pytest shard is the floor тАФ further reduction needs PERF-1 (pytest internal parallelism).

### release.yml

- `ruff`: ~1 min (parallel, sub-30s warm)
- `pytest` longest shard (4-way): ~10-15 min (was ~35-45 min on single runner)
- `build`: ~3-5 min (unchanged)
- `publish-pypi`: ~2-3 min (unchanged)
- `github-release`: ~1-2 min (unchanged)

**Total wall time = max(ruff, pytest) + build + publish + release**
= max(~1, ~10-15) + 5 + 3 + 2
= **~20-25 min**

vs. before **~45-55 min**.

**Reduction: ~50-55%** on release.yml. **Goal met.**

## Honest limits

- These are **estimates**, not measurements. Actual numbers depend on GHA free-runner queue depth + cache hit rate. First post-merge run will be cold cache тЖТ expect ~+3-5 min on warm-cache estimates above.
- `mypy` `continue-on-error: true` preserved per CLAUDE.md "best effort treat new errors as red" тАФ _not_ skipped, but doesn't fail the PR until stabilized.
- `pytest` floor is the slowest shard (~10-15 min for the heaviest shard with DB fixtures). PERF-1 (pytest internal parallelism via `pytest-xdist`) is the next lever once this lands.
- `pip-audit` will surface CVEs that were previously _silently swallowed_ on PRs where `pytest` shard 0 failed before reaching pip-audit. Possible new yellow lights тАФ by design, `continue-on-error: true` so it doesn't fail PR.
