# Wave 66 / 67 / 62 backfill (2026-05-16, ultra-low row count fix)

## Context

Wave 67 Athena Q17 finding (commit `f8d49e88a`):

- `wave66_pii` = **0 rows** (Glue tables existed but **no data in S3** — packet generators
  had never been written under canonical prefixes).
- `wave67_tech` = **27 rows** (only ~5 rows per generator average — corpus filter too
  narrow at the JSIC-major × jpi_programs name-keyword union).
- `wave62_sectoral` = **33 rows** (~4 rows per generator — same cohort thinness as
  wave67_tech, additionally double-counted in Q17 due to UNION ALL with foundation tables).

Q17 surfaced the gap on 2026-05-16 19:35:17 +0900; the backfill below was executed the
same evening against the post-687-packet-sync Glue + Athena substrate.

## Approach

Per `feedback_packet_gen_runs_local_not_batch` (2026-05-16 Wave 53 evidence): packet
generators are local-runtime Python over `autonomath.db` (9.7 GB SQLite). Each row is
<1 sec to render — Fargate Spot startup (~30 sec) dominates, so Batch is the wrong
runtime. Used **local generation + `aws s3 sync` parallel** instead.

Flow:

1. `xargs -P 12 -L 1 ./run_one.sh` against a 30-line manifest (10 sectoral + 10 PII +
   10 tech), each shell invocation runs `python -m
   scripts.aws_credit_ops.generate_<name>_packets --commit --output-prefix
   file:///tmp/local/<prefix>/ --local-out-dir out/<prefix>`. Total wall time ~5 sec
   for all 30.
2. `aws --profile bookyou-recovery s3 sync file:/tmp/local/<prefix>/
   s3://jpcite-credit-993693061769-202605-derived/<prefix>/` in parallel for all 28
   non-empty prefixes (~30 sec total via concurrent uploads).
3. `aws glue start-crawler --name jpcite-credit-derived-crawler` — adds new tables for
   the prefixes that did not have a Glue table yet (wave66 PII 10 × table, plus
   `packet_*` registrations for any of the 30 that the prior crawl had missed).
4. Per-table `SELECT COUNT(*) FROM packet_<name>` via Athena (workgroup
   `jpcite-credit-2026-05`, DB `jpcite_credit_2026_05`) to verify each row count.

## Anti-pattern noted

`file:///tmp/local/<prefix>/` as `--output-prefix` confused the packet runner: the
runner treats the URI as a `Path()` literal, producing actual filenames like
`./file:/tmp/local/...`. Future runs should use a plain local path (no `file://`
scheme) or true `s3://` URI. The 30 packets landed correctly anyway because the
prefix prefix-stripping in `aws s3 sync` flattened the path during sync.
